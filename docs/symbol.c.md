## 模块: symbol.c — 符号表与常数表实现

## 简明解释

编译器原理：符号表是连接所有编译阶段的中心数据结构——词法分析发现标识符后存入符号表，语义分析填充类型、偏移等属性，代码生成阶段查询符号表以确定变量分配。

各函数简明解释：

- **`sym_init`** — 清零所有表与计数器：`sym_count`、`const_count`、`temp_count`、`label_count`、`sem_top` 归零；`cur_type` 设为 `TY_INTEGER`；`cur_offset` 归零；`memset` 清空 `sym_table[]`、`const_table[]`、`sem_stack[]`。每次编译新源文件前调用一次，以建立纯净的初始状态。
- **`sym_enter_id`** — 线性搜索 O(n)，在 `sym_table[]` 中查找同名标识符，命中则返回已有索引，未命中则在尾部插入（先检查 `MAX_SYMBOLS` 上限）。用 `strncpy` 安全复制名称，用 `type_len()` 自动计算类型宽度。方法简单但适用于教学编译器的小规模程序。
- **`sym_lookup_id`** — 纯搜索函数：遍历 `sym_table[]` 比较名称，命中返回索引，无命中返回 `-1`。不作任何插入，用于语义分析阶段引用变量时查找其定义。
- **`sym_set_type`** — 翻译文法 a6 动作的核心：将 `sym_table[index]` 的 `type`、`len`、`offset` 覆盖更新为实参值。在变量声明文法 `var ID_SEQ : TYPE ;` 中，标识符在 a2 阶段以默认类型填入，到 a6 阶段才按序分配真实偏移并修正类型——`sym_set_type` 承担该覆盖写入。
- **`sym_enter_const`** — 常数查填：线性搜索 `const_table[]`，用 `fabs(a - b) < 1e-12` 进行浮点容差比较（双精度机器精度约 `2.22e-16`，`1e-12` 可容忍舍入误差同时作为安全余量）。命中返回索引，未命中则插入尾部。整型常量已统一提升为 `double`。
- **`sym_lookup_const`** — 常数纯查询：同样的 `1e-12` 容差比较，但不插入条目，未找到返回 `-1`。
- **`sym_new_temp`** — 递增 `temp_count`，生成名称 `"t1"`、`"t2"`…，调用 `sym_enter_id` 以 `KIND_TEMP` 登记。返回 `char *` 指向函数内 `static` 缓冲区——每次调用会覆盖上次返回值，调用方必须在使用前复制到独立存储（本项目在生成四元式时立即 `strncpy` 到 `Quadruple` 结构体的字段中）。
- **`sym_new_label`** — 极简实现：仅执行 `++label_count` 并返回。标号以纯数字形式存在四元式的 `result` 字段，不需要名字存储或查填。
- **`sym_dump`** — 遍历 `sym_table[]`，用 `snprintf` 渐进累积格式化输出：表头（索引/名字/种类/类型/偏移/宽度）、分隔线、每行数据。种类用三元运算符映射为中文（程序/变量/临时），类型通过 `type_names[]` 数组转译。
- **`const_dump`** — 与 `sym_dump` 同模式：遍历 `const_table[]`，输出表头（索引/值）和每行的 `%g` 格式数值。注意两个 dump 函数各自从 `buf[0]` 写入，连续调用需手动管理偏移量。

## 补充专题讲解

### 1. `sym_init()` — 每个字段重置的含义

`sym_init()` 被调用后，`Compiler` 上下文进入"空状态"，等同于刚创建未经使用的编译器实例。各字段含义如下表：

| 字段 | 初始值 | 含义 |
|------|--------|------|
| `sym_count` | 0 | 符号表中已登记条目数为 0。后续 `sym_enter_id()` 每次插入一条记录，该值加 1。同时也是"下一个可用槽位"的索引。 |
| `const_count` | 0 | 常数表中已登记条目数为 0。与 `sym_count` 角色对称。 |
| `temp_count` | 0 | 临时变量编号计数器。第一次调用 `sym_new_temp()` 时先自增为 1，生成 `t1`。 |
| `label_count` | 0 | 标号编号计数器。第一次调用 `sym_new_label()` 时先自增为 1，标号为 L1。 |
| `sem_top` | 0 | 语义栈栈顶指针，指向 `sem_stack[]` 中下一个空位。0 表示栈为空。a2 语义动作向栈中压入符号表索引，`sem_top` 递增。 |
| `cur_type` | `TY_INTEGER` (0) | 当前正在声明的变量类型默认值。在解析器遇到 `var` 后、`:` 之前，尚未确定实际类型，暂用整数类型作为占位。遇到 `: real` 或 `: char` 时由 a3 动作更新。 |
| `cur_offset` | 0 | 栈帧偏移的起点。每声明一个变量，偏移值加上该变量类型的 `len`（宽度），形成紧凑的连续内存布局。例如先声明 `a: integer`（偏移 0→宽度 4→新偏移 4），再声明 `b: real`（偏移 4→宽度 8→新偏移 12）。 |
| `sym_table[]` | 全 0 (memset) | 物理清零。所有 `Symbol` 的 `name` 为空字符串（`'\0'`）、`type=0`（即 `TY_INTEGER`）、`kind=0`（即 `KIND_PROGRAM`）、`offset=0`、`len=0`。 |
| `const_table[]` | 全 0.0 (memset) | 物理清零。所有常数值槽位初始化为 `0.0`。 |
| `sem_stack[]` | 全 0 (memset) | 物理清零。栈槽位全部为 0（0 是合法的符号表索引，但配合 `sem_top=0` 表示栈为空，不会访问这些槽位）。 |

### 2. `sym_enter_id()` — 线性搜索 + 插入逻辑详解

```
sym_enter_id(c, name, type, kind, offset):

    ┌─ for i = 0 to sym_count-1 ────────────┐
    │   strcmp(sym_table[i].name, name)==0 ? │
    │     YES → return i  (命中，直接返回)    │
    │     NO  → 继续循环                      │
    └────────────────────────────────────────┘
    │ (循环结束，未找到)
    ▼
    ┌─ sym_count >= MAX_SYMBOLS ? ──────────┐
    │   YES → return -1  (表满，报错)         │
    │   NO  → 继续插入                        │
    └────────────────────────────────────────┘
    ▼
    i = sym_count++     (取尾索引，计数+1)
    拷贝 name → sym_table[i].name
    填入 type, kind, offset → sym_table[i]
    len = type_len(type)
    return i
```

**关键语义**：
- **查填合一**：相同的标识符不管出现多少次，在符号表中只有一个条目。这确保了后续语义分析中对同一变量的引用指向同一个 `type` 和 `offset`。
- **命名区分**：虽然是简单线性搜索，但 C 语言的 `strcmp` 区分大小写，所以 `MyVar` 和 `myvar` 是两个不同的标识符。
- **越界保护**：`strncpy` + `MAX_NAME - 1` 防止超长标识符溢出，`MAX_SYMBOLS` 检查防止条目数越界。

### 3. `sym_set_type()` — 翻译文法 a6 语义动作的配套函数

在变量声明的翻译文法中，语义动作的执行顺序如下：

```
变量声明文法： var ID_SEQ : TYPE ;
语义动作编号： a1  a2  ... a3  a4  a5  a6
```

| 动作 | 位置 | 行为 | sym_set_type 的角色 |
|------|------|------|---------------------|
| a1 | 读到 `var` | 设置 `cur_type` 为默认值（整数） | 不涉及 |
| a2 | 读到一个 `id` | 将标识符登记到符号表（用 a1 的类型），**并将索引压入语义栈** | 不涉及 |
| a2 | 读到 `,` 后的 `id` | 同上，继续压栈 | 不涉及 |
| a3 | 读到 `:` | 暂存当前 `cur_offset` 值，为后续计算准备 | 不涉及 |
| a4 | 读到 `integer` / `real` / `char` | 更新 `cur_type` 为实际声明的类型 | 不涉及 |
| a5 | —（无动作） | — | — |
| **a6** | 读完整个声明 | 从语义栈中按 **FIFO** 顺序弹出标识符索引，逐个计算偏移并更新 | **核心调用者** |

a6 动作的伪代码：

```
for each id_index popped from sem_stack (in FIFO order):
    c->cur_offset += type_len(c->cur_type)  // 或先加后赋，取决于具体实现
    sym_set_type(c, id_index, c->cur_type, type_len(c->cur_type), c->cur_offset)
```

`sym_set_type()` 在这里的作用是：将 a1/a2 阶段用默认类型填入的符号表条目，**覆盖更新**为实际声明的类型、宽度和正确的偏移地址。因为 a2 动作在读到标识符时就立即填入了符号表，但那时还不知道类型和偏移，所以需要 a6 阶段批量修正。

### 4. `sym_enter_const()` — 为什么容差值选择 `1e-12`

常数比较使用 `fabs(a - b) < 1e-12` 而不是 `a == b`，原因有三：

1. **IEEE 754 浮点误差**：`double` 类型使用 64 位 IEEE 754 双精度表示，约 15-17 位有效十进制数字。同一个数学上的数值在经过不同运算路径后可能产生微小的舍入误差（如 `0.1 + 0.2` 的经典案例），直接用 `==` 比较会错误地判定为不相等。

2. **`1e-12` 的选择依据**：双精度浮点数的机器精度（machine epsilon）约为 `2.22e-16`。用 `1e-12` 作为容差，比机器精度大约 4 个数量级，既能容忍大多数舍入误差，又对于教学编译器中的典型数值范围（±1e6 量级）足够精确。12 位小数的精度远超 Pascal 教学语言中 `real` 类型的需求。

3. **常数合并安全**：在编译器前端，同一个数学常数（如 `3.14`）无论在源码中出现多少次，都应该合并为常数表中的同一个条目。使用容差比较确保了这种合并的鲁棒性。

**潜在问题**：如果两个常数差距非常小（如 `1.0000000000005` 和 `1.0000000000006`），它们会被错误地判定为相等。但在教学编译器的使用场景中，源程序中的常数几乎都是字面量（如 `42`、`3.14`），不会出现这种情况。

### 5. `sym_new_temp()` — `static buf` 的设计隐患

```c
char *sym_new_temp(Compiler *c) {
    static char buf[MAX_NAME];   // ← 关键
    c->temp_count++;
    snprintf(buf, MAX_NAME, "t%d", c->temp_count);
    sym_enter_id(c, buf, c->cur_type, KIND_TEMP, c->cur_offset);
    return buf;
}
```

**`static` 局部变量的行为**：
- 存储在静态数据区（.data 或 .bss 段），而非栈上。
- 整个程序生命周期只有 **一份实例**。
- 所有调用 `sym_new_temp()` 的地方共享同一块 `buf` 内存。

**隐患场景**：

```c
char *t1 = sym_new_temp(c);   // t1 指向 buf，buf 内容为 "t1"
char *t2 = sym_new_temp(c);   // t2 指向 buf，buf 内容为 "t2"
// 此时 t1 和 t2 指向同一块内存！
// t1 实际读到的内容是 "t2"
printf("%s %s\n", t1, t2);    // 输出: "t2 t2"
```

**为什么会出问题**：当代码生成时需要同时持有多个临时变量名时（例如生成 `t1 := t2 + t3` 这样的四元式），如果前一个 `sym_new_temp()` 返回的指针还未使用完毕就再次调用，返回的指针所指向的内容已经被覆盖了。

**规避方式**（本项目的做法）：在四元式生成中，每次调用 `sym_new_temp()` 后立即用 `strncpy` 将名称复制到 `Quadruple` 结构体的 `arg1`/`arg2`/`result` 字段中（这些字段是 `char[MAX_STR]` 数组，独立的副本）。因此虽然指针不稳定，但拷贝后的副本是安全的。这是一种"使用时即刻复制"的模式。

**更好的设计**：修改接口为 `void sym_new_temp(Compiler *c, char *out_buf)`，由调用方提供缓冲区，避免共享静态内存。

### 6. `sym_new_label()` — 极简设计

```c
int sym_new_label(Compiler *c) {
    return ++c->label_count;
}
```

这是一个只做计数器递增的函数，没有在符号表或常数表中登记任何内容。为什么这样就够了？

**标号不需要名字存储**：标号在四元式中以数字形式存在（存储在 `Quadruple` 的 `result` 字段），生成目标代码时再格式化为 `L1`、`L2` 等标签。它们是编译期内部使用的跳转目标，不需要像变量名那样进行符号解析。

**编号连续性**：每次调用返回一个严格递增的唯一整数，确保跳转指令能正确配对（`OP_JMP` 的 `result` 字段 = 目标 `OP_LABEL` 的 `result` 字段）。

### 7. `sym_dump()` 和 `const_dump()` — 输出格式示例

**`sym_dump()` 典型输出**：

```
========== 符号表 (Symbol Table) ==========
索引   名字               种类       类型       偏移   宽度
------ ---------------- -------- -------- ------ ----
0      main             程序       integer  0      4
1      x                变量       integer  4      4
2      y                变量       real     8      8
3      t1               临时       integer  16     4
```

**格式要点**：
- 索引列宽 6，左对齐（`%-6d`）
- 名字列宽 16，左对齐（`%-16s`），超出 16 字符会被截断（但 `MAX_NAME=32`，实际名字 ≤ 31 字符，可能超出列宽）
- 种类和类型列宽 8，左对齐（`%-8s`）
- 偏移列宽 6，左对齐（`%-6d`）
- 宽度列无固定宽度，右对齐（`%d`）

**`const_dump()` 典型输出**：

```

========== 常数表 (Constant Table) ==========
索引   值
0      3.14
1      42
2      2.71828
3      1e-10
```

**格式要点**：
- 以空行（`\n` 开头）与前面内容分隔
- 索引列宽 6，左对齐（`%-6d`）
- 值用 `%g` 格式输出，自动选择最短表示（`42.0`→`42`，`3.14`→`3.14`，`1e-10`→`1e-10`）

**注意事项**：`sym_dump()` 和 `const_dump()` 都从 `buf[0]` 开始写入（`pos = 0`），即各自覆盖缓冲区的起始位置。如果调用方想将两者连续输出，需要：
