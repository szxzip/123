## 模块: scanner.c — 词法分析器实现

## 简明解释

**词法分析器是编译系统的第一道关卡**，负责将源程序字符流转换为结构化的 Token 序列，供语法分析器消费。scanner.c 采用 DFA 系统化方法，将关键字/界符识别、标识符识别、常数识别统一在 `scanner_next_token()` 的三大分支中。

### 各组件要点

- **`keys_table[]`**（35 条目含 NULL 哨兵）：关键字与界符的统一查找表。`reserve()` 在此表中线性搜索，匹配成功返回 `i+3`（恰好映射 `TOK_PROGRAM=3` 到 `TOK_WRITE=35`），返回 0 表示标识符，调用者据此分流。
- **`const_aut[8][5]`**：Pascal 常数 DFA 状态转换矩阵——8 个状态 × 5 列输入（digit/dot/E/±/other）。状态含义：1=初态（必须以数字开头）、2=整数态（累加 `n=10n+d`）、3=刚读点（标记小数开始）、4=小数态（累加 `n=10n+d, m++`）、5=刚读 E（标记指数开始）、6=刚读符号（记录 `e=±1`）、7=指数态（累加 `p=10p+d`）、8=接受态（计算最终值）。识别示例：`123` 走 1→2→2→2→8，`3.14` 走 1→2→3→4→4→8，`1.5E+3` 走 1→2→3→4→5→6→7→8。
- **`next_char()` / `skip_whitespace()`**：逐字符推进扫描器，分别负责消费一个字符和跳过空格/制表/换行。`next_char()` 在到达文件尾时将 `c->ch` 设为 `'\0'` 作为 EOF 哨兵。
- **`reserve()`**：对 `keys_table[]` 做 O(n) 线性搜索，利用 NULL 哨兵终止。返回 `i+3` 直接可用作 Token 码，返回 0 触发标识符分流。设计精妙：`reserve()` 返回 0 既是 TOK_EOF 的值又是"未找到"标记，但调用上下文已先行拦截 EOF，不会产生二义性。
- **`constant_automaton()`**：完整的 DFA 循环——每轮确定当前字符的列号（digit=0, dot=1, E/e=2, ±=3, other=4），查 `const_aut[s-1][col]` 获取新状态，执行对应语义动作。状态 8 触发时 `break` 但不消费触发字符（正确保留给下一个 Token）。最终公式 `num = n × 10^(e×p − m)` 从累积尾数 `n`、小数位数 `m`、指数符号 `e`、指数值 `p` 还原实际浮点值。
- **`scanner_next_token()` 三大分支**：① `is_letter(ch)` → 循环读字母/数字到 `str_token`，`reserve()` 查表，返回 0 则设为 `TOK_ID` 并调用 `sym_enter_id()` 入符号表；② `is_digit(ch)` → 调用 `constant_automaton()`，设为 `TOK_CONST` 并调用 `sym_enter_const()` 入常数表；③ 其他 → 先处理 `:=` `<=` `>=` `<>` 双字符向前看，再查 `keys_table`，未找到则报"未识别的符号"错误。
- **`scanner_scan_all()` / `scanner_dump_tokens()`**：批量扫描——`scanner_scan_all()` 重置扫描器后循环调用 `scanner_next_token()` 填充 `token_list[]`，末尾追加 EOF 哨兵。`scanner_dump_tokens()` 遍历 `token_list[]` 输出格式化结果：标识符 `(k,1)(i,N)`、常数 `(c,N)`、关键字/界符 `(p,N)`，每 8 个换行。

### 编译器原理

词法分析是编译器前端的第一阶段，其任务是将源程序的字符序列转化为有结构的 Token 序列——每个 Token 包含类别码（code）和可选的属性值（value，如符号表/常数表索引）。DFA 方法将识别过程抽象为状态转移表，每类词法规则对应一条 DFA 路径，实现方式简洁、系统化且易于验证。Pascal 常数自动机是这一思想的典型体现：8 个状态覆盖整数、实数、科学计数法三种格式，5 列输入分类穷举所有可能字符，状态转移矩阵直接驱动识别过程。

> **为什么 code = index + 3？**
>
> Token 码的分配遵循先特殊后通用的原则：
> - `TOK_EOF = 0`：文件结束，特殊的哨兵 Token
> - `TOK_ID = 1`：标识符，所有用户自定义名字的统一类别
> - `TOK_CONST = 2`：常数，所有字面常数的统一类别
> - `TOK_PROGRAM = 3` 起：关键字和界符，从 3 开始编号
>
> 因此关键字/界符表中索引从 0 开始，加 3 后正好映射到 `grammar.h` 中的 `TOK_PROGRAM` 及后续枚举值。`reserve()` 函数返回 `i + 3`，扫描器根据返回值是否为 0 判断是标识符还是保留字——0 表示未查到（标识符），非 0 表示是关键字/界符且值就是 Token 码。

### reserve() 的 index+3 映射原理

```
grammar.h 中的 Token 码枚举:
  TOK_EOF      = 0   ─── 不在 keys_table 中，reserve 不会返回 0 以外的此值
  TOK_ID       = 1   ─── 标识符，由调用于 reserve 返回 0 时分流处理
  TOK_CONST    = 2   ─── 常数，由常数自动机单独识别
  TOK_PROGRAM  = 3   ─── keys_table[0]="program"  → reserve 返回 0+3=3 ✓
  TOK_VAR      = 4   ─── keys_table[1]="var"      → reserve 返回 1+3=4 ✓
  TOK_INTEGER  = 5   ─── keys_table[2]="integer"  → reserve 返回 2+3=5 ✓
  ...                  以此类推...
  TOK_WRITE    = 35  ─── keys_table[32]="write"   → reserve 返回 32+3=35 ✓
```

**设计巧妙之处**：`reserve()` 返回 0 既表示"未找到"又恰好是 TOK_EOF 的值，但调用上下文（`scanner_next_token()` 第 150 行）明确区分：如果同时满足"非 EOF 状态"且"reserve 返回 0"，则判定为标识符。EOF 状态在 `scanner_next_token()` 开头（第 133 行）就已经被特判返回了，根本不会走到 `reserve()` 调用，因此不会产生二义性。

### scanner_next_token() 整体决策流程

```
scanner_next_token(c)
  │
  ├─ skip_whitespace(c)                    ← 总是第一步
  │
  ├─ if ch == '\0' → TOK_EOF, return       ← EOF 检查
  │
  ├─ if is_letter(ch) ───────────────────────→ 标识符/关键字
  │    ├─ 循环读取字母数字到 str_token
  │    ├─ code = reserve(str_token)
  │    │    ├─ code == 0 → TOK_ID + sym_enter_id()
  │    │    └─ code != 0 → 直接使用 code
  │    └─ return
  │
  ├─ if is_digit(ch) ────────────────────────→ 常数
  │    ├─ val = constant_automaton(c)
  │    ├─ code = TOK_CONST
  │    ├─ value = sym_enter_const(val)
  │    └─ return
  │
  └─ else ───────────────────────────────────→ 界符
       ├─ 特殊处理 ':=' '<=' '>=' '<>' (向前看)
       ├─ 其他字符直接消费
       ├─ code = reserve(str_token)
       └─ return
```

| 245 | `"(k,1)(i,%d) ", t->value);` | **标识符输出格式**：`(k,1)(i,value)` —— `k` 表示种类（1=标识符），`i` 表示符号表索引（`t->value`）。例如 `(k,1)(i,3)` 表示第 3 号符号表条目是一个标识符。 |
| 246 | `else if (t->code == TOK_CONST)` | 判断 Token 类型是否为常数（`TOK_CONST = 2`）。常数显示格式为 `(c,索引)`。 |
| 247 | `pos += snprintf(buf + pos, bufsize - pos,` | 同上。 |
| 247 | `"(c,%d) ", t->value);` | **常数输出格式**：`(c,value)` —— `c` 表示常数，`value` 是常数表索引。例如 `(c,2)` 表示第 2 号常数表条目。 |
| 248 | `else` | 非标识符也非常数——即关键字或界符（`TOK_PROGRAM` ~ `TOK_WRITE`）。 |
| 249 | `pos += snprintf(buf + pos, bufsize - pos,` | 同上。 |
| 249 | `"(p,%d) ", t->code);` | **关键字/界符输出格式**：`(p,code)` —— `p` 表示 Pascal 保留字/界符，`code` 是 Token 码。例如 `(p,3)` 表示 `TOK_PROGRAM`=3，即 `program` 关键字。 |
| 250 | `if ((i + 1) % 8 == 0)` | 每输出 8 个 Token 后换行，保持输出整洁。`(i+1) % 8 == 0` 等价于"第 8、16、24... 个 Token 之后"。 |
| 251 | `pos += snprintf(buf + pos, bufsize - pos, "\n");` | 写入换行符。 |
| 252 | `}` | for 循环结束。 |
| 253 | `pos += snprintf(buf + pos, bufsize - pos, "\n");` | 循环结束后追加一个最终换行符，确保输出以换行结束。 |
| 254 | `}` | 函数结束。 |

### scanner_dump_tokens() 输出示例

假设源程序为：
```pascal
program test; var x: integer; begin x := 123 end.
```

输出：
```
========== Token 序列 (词法分析输出) ==========
(p,3) (k,1)(i,0) (p,12) (p,4) (k,1)(i,1) (p,11) (p,5) (p,12)
(p,8) (k,1)(i,1) (p,13) (c,0) (p,9) (p,18)
```

解释：
- `(p,3)` → `program` 关键字
- `(k,1)(i,0)` → 标识符 `test`，符号表索引 0
- `(p,12)` → `;`
- `(p,4)` → `var`
- ...等等

## 附录：全局数据流

```
源程序文件 (.txt)
  │
  ▼
main.c: 读取文件内容为 C 字符串
  │
  ▼
scanner_init(c, source)       ← 初始化 (source, len, pos, ch)
  │
  ▼
scanner_next_token(c)         ← 逐 Token 识别
  │  ├─ skip_whitespace:      跳过空白
  │  ├─ 标识符:               is_letter → 读字母数字 → reserve() → TOK_ID + sym_enter_id()
  │  ├─ 常数:                 is_digit  → constant_automaton() → TOK_CONST + sym_enter_const()
  │  └─ 界符:                 查 keys_table (':=' '<=' '>=' '<>' 需向前看)
  │
  ▼
scanner_scan_all(c)
  └─ 循环 scanner_next_token → 填充 token_list[]
  │
  ▼
scanner_dump_tokens(c, buf)
  └─ 格式化输出: (k,1)(i,N) / (c,N) / (p,N)
```

## 附录：关键设计决策总结

| 决策 | 说明 |
|------|------|
| **keys_table 统一表** | 关键字和界符存储在同一数组中，使用 index+3 映射到 Token 码。`reserve()` 返回 0 表示标识符，调用者据此分流。 |
| **自定义 is_letter/is_digit** | 不使用 `<ctype.h>`，避免 locale 影响，直接 ASCII 范围比较，简洁精确。 |
| **Pascal 常数自动机** | 8 状态 DFA 完整覆盖整数、实数、科学计数法。5 列分类（digit/dot/E/+-/other），语义动作采用累计式算法（n 累积数字，m 计算位数，公式 `num = n × 10^(e×p - m)`）。状态 8 触发时不消费当前字符，正确保留后续 Token 的起始字符。 |
| **双字符界符向前看** | `:=` `<=` `>=` `<>` 通过对首字符（`:` `<` `>`）特殊判断，向前看一个字符来确定。非双字符的情况（如 `:` 后不是 `=`），已读的后一字符保留在 `c->ch` 中作为下一个 Token 的起始。 |
| **两遍扫描设计** | `scanner_scan_all()` 在第一遍完成全部 Token 识别和符号/常数表填充；语法分析阶段重新初始化并逐 Token 消费（同时可随机访问 `token_list[]` 向前看）。 |
